## 🔄 HOW TO USE THIS SYSTEM

### Setup (15 minutes)

1. **For AI Planning Agent:**
   - Copy complete text of `SYSTEM_PROMPT.md`
   - Paste as system prompt in your agent platform
   - The agent now has complete architectural guidance

2. **For Human Stakeholders:**
   - Share `PLANNING_WORKFLOW_GUIDE.md` with your team
   - Explain the Three-Phase model (5 min)
   - Answer any questions about the process
   - Clarify that this is _planning_, not implementation

3. **For Implementation Teams:**
   - Save `QUICK_REFERENCE_TEMPLATES.md` for future reference
   - Prepare to receive 15-20 markdown documents as final SDD

### Engagement (2-4 hours total)

**Phase 1: Intake & Gap Analysis** (30-60 minutes)
```
You: "I want to build a real-time collaborative editor"
                    ↓
Agent: Performs gap analysis, asks 5-7 clarifying questions
                    ↓
You: Answer questions or approve baseline assumptions
                    ↓
✅ APPROVAL GATE 1: Assumptions documented and approved
```

**Phase 2: Blueprint Proposal** (45-90 minutes)
```
Agent: Presents three-block architecture (Frontend, Backend, Infrastructure)
       Explains technology choices and rationale
       Justifies against your constraints
                    ↓
You: Review blueprint using QUICK_REFERENCE_TEMPLATES.md checklist
     Ask questions, request modifications, or approve
                    ↓
✅ APPROVAL GATE 2: Blueprint explicitly approved
```

**Phase 3: SDD Generation** (60-120 minutes)
```
Agent: Generates 15-20 markdown documents:
       - Executive summary & assumptions
       - Architecture patterns & design decisions
       - Frontend, Backend, Database specifications
       - Infrastructure, Security, Compliance details
       - API contracts & data models
       - Testing hierarchy & CI/CD pipeline
       - Deployment procedures & monitoring
       - SOLID principle justifications
       - Risk assessment & open questions
                    ↓
You: Review final package for completeness
                    ↓
✅ APPROVAL GATE 3: SDD approved, ready for implementation
```

### Handoff (Immediately)

```
Implementation teams receive:
- 15-20 complete markdown documents
- Specific technology names (not categories)
- API contracts and data schemas
- Deployment and testing procedures
- SOLID principle guidance
- Risk assessment and mitigations

Teams begin implementation with zero architectural ambiguity
```

## 🎯 KEY FEATURES OF THIS SYSTEM

### 1. **Bounded Technology Catalogs** (Prevents Hallucinations)
Instead of generic guidance like "use a database," the system provides:
- 7 specific database types with strategic purposes
- 8 frontend frameworks with exact use cases
- 5 backend languages with optimal scenarios
- 4 cloud platforms with integration specifics

Each recommendation is grounded in concrete 2026 technology, not vague categories.

### 2. **Three Explicit Approval Gates** (Ensures Stakeholder Alignment)
- **Gate 1:** Assumptions explicitly documented and signed off
- **Gate 2:** Blueprint reviewed and approved before detailed design
- **Gate 3:** Final SDD reviewed for completeness before implementation

No proceeding to next phase without explicit human approval.

### 3. **SOLID Principles Enforcement** (Ensures Quality)
SOLID principles (Single Responsibility, Open/Closed, Liskov Substitution, Interface Segregation, Dependency Inversion) are explicitly enforced across:
- Service boundaries
- Component hierarchies
- API contracts
- Database schemas
- Infrastructure layers

Not just aspirational guidance—actually verifiable and reviewable.

### 4. **Production-Ready Output** (Not Theoretical)
The final System Design Document contains:
- ✅ Specific technologies (PostgreSQL, not "a database")
- ✅ API contracts (OpenAPI specifications)
- ✅ Component boundaries (exact file/service structure)
- ✅ Deployment procedures (step-by-step Canary or Blue-Green)
- ✅ Testing requirements (specific coverage targets)
- ✅ Infrastructure code templates (Terraform, IaC)
- ✅ Security configurations (WAF rules, encryption specifics)

**Not:** Generic guidance, category recommendations, theoretical frameworks

### 5. **Risk-Aware Planning** (Acknowledges Reality)
Every architecture includes:
- Identified risks with impact assessment
- Mitigation strategies
- Known unknowns and learning opportunities
- Technology tradeoffs explicitly documented
- Operational complexity acknowledgment

## 📞 SUPPORT

### Common Questions

**Q: How long does a complete planning engagement take?**
A: 2-4 hours total (Phase 1: 30-60 min, Phase 2: 45-90 min, Phase 3: 60-120 min)

**Q: Can we skip Phase 2?**
A: No. Phase 2 is your last chance to review architecture before detailed design. Skipping it causes problems.

**Q: What if we need to change architecture during implementation?**
A: Document the reason in the SDD's decisions log. Major changes should trigger a Phase 2 revision.

**Q: Can we use this for existing legacy systems?**
A: Yes, excellent for planning migrations. Will help systematically decompose legacy monoliths.

**Q: Is this suitable for small teams?**
A: Yes, adjust complexity appropriately. A 3-person startup SDD is simpler than enterprise, but follows same framework.
